# Ops Tool 前端版需求文档

## 1. 文档说明

本文档用于明确 `ops_tool` 前端版的第一阶段需求，作为后续实现计划和开发执行的依据。

当前约定：

- 先整理需求并确认
- 暂不提交到 GitHub
- 等工具开发完成并确认最终版本后，再统一整理并提交

## 2. 项目目标

基于现有脚本 [sitemap_parser.py](<D:\my-tools\ops_tool\scripts\sitemap_parser.py>)，开发一个带前端界面的本地工具，让用户不需要使用命令行也能完成同样的处理流程。

第一阶段目标不是做成复杂系统，而是优先做成一个：

- 本地可用
- 操作简单
- 能快速投入实际使用
- 保留现有脚本能力

## 3. 产品定位

这是一个本地单机使用的轻量运营工具，不是多人协作平台，也不是在线 SaaS 系统。

工具核心用途是：

- 读取网站 sitemap
- 读取本地 Word 文档
- 可选读取关键词配置文件
- 调用现有 Python 逻辑完成处理
- 生成 AI 可直接使用的 prompt
- 在页面中展示结果并支持复制

## 4. 目标用户

主要用户为内部运营人员或工具使用者，特点是：

- 希望少用命令行
- 更习惯表单和按钮式操作
- 希望错误信息更直观
- 希望一个任务运行时还能继续把后续任务放进去排队

用户不需要理解代码实现，但应该能看懂工具当前在做什么、哪里成功、哪里失败。

## 5. 第一阶段范围

### 5.1 包含范围

- 本地网页界面
- 单页面操作方式
- 本地选择 Word 文件
- 本地选择关键词配置文件
- 输入网站域名
- 选择 CMS 模式
- 创建任务
- 顺序执行任务队列
- 在一个任务运行时继续添加新任务
- 显示任务状态
- 显示每个任务的完整日志
- 显示每个任务的结果摘要
- 显示每个任务生成的 prompt
- 每个任务的 prompt 都支持单独复制
- 显示输出目录或输出结果位置
- 复用当前 Python 脚本的核心处理能力

### 5.2 暂不包含

- 多个任务并行执行
- 用户登录
- 云端同步
- 多人协作
- 远程部署
- 复杂权限系统
- 数据库持久化系统
- 对现有 prompt 规则做大改

## 6. 成功标准

第一阶段完成后，工具应达到以下目标：

1. 用户打开页面后，不需要命令行知识也能提交任务
2. 用户可以直接选择本地 Word 文件，不需要手输路径
3. 用户可以在一个任务运行时继续添加后续任务
4. 每个任务完成后，都能看到摘要、完整日志、prompt 和复制按钮
5. prompt 可直接复制使用
6. 失败任务不会卡住整个队列，后续任务仍能继续执行

简单说，首版的成功标准是：

`一个不看说明的人，也能大概率自己完成第一次任务提交和结果复制。`

## 7. 核心使用流程

### 7.1 正常流程

1. 用户打开本地网页
2. 选择一个或多个本地 Word 文件
3. 可选选择关键词配置文件
4. 输入网站域名
5. 选择 CMS 模式
6. 点击开始任务
7. 任务进入队列
8. 如果当前没有运行中的任务，则立即开始执行
9. 如果当前已有运行中的任务，则进入等待中
10. 用户可以继续添加其他任务
11. 用户点击某个任务后，可以查看该任务详情
12. 任务完成后，页面展示：
    - 结果摘要
    - 完整日志
    - prompt 内容
    - 复制按钮
    - 输出目录或结果位置

### 7.2 异常流程

1. 如果任务在 sitemap 抓取、Word 转换、关键词处理或 prompt 生成阶段出错
2. 任务状态显示为失败
3. 页面展示完整日志和明确的失败原因
4. 队列自动继续执行下一个等待中的任务

## 8. 易用性要求

这个工具的重点之一是“足够简单”，因此首版必须满足以下要求：

- 采用单页面，不拆成多个操作页面
- 文件通过本地文件选择器选择，不要求手输路径
- 运行中任务和等待中任务要明显区分
- 结果区里复制按钮要明显，不要藏得太深
- 日志要能直接看，不要求下载后查看
- 默认尽量记住上一次输入的域名和 CMS 模式

首版对“简单易用”的定义是：

- 一次正常任务只需要少量操作
- 用户看得懂当前状态
- 出错时知道问题出在哪一步

## 9. 页面结构要求

页面建议分为三个主要区域：

1. 新建任务区
2. 任务队列区
3. 任务详情区

### 9.1 新建任务区

包含字段：

- 网站域名
- CMS 模式
- Word 文件选择
- 关键词配置文件选择
- 开始任务按钮

要求：

- 清楚显示已选文件名
- 域名和 CMS 尽量记住上次使用值
- 没选 Word 文件或没填域名时，不允许提交

### 9.2 任务队列区

每个任务需要显示：

- 任务名称或任务标题
- 当前状态
- 创建时间
- 简短进度说明

任务状态包括：

- 等待中
- 运行中
- 成功
- 失败

首版只允许同时存在一个运行中任务。

### 9.3 任务详情区

用户选中某个任务后，需要显示：

- 结果摘要
- 完整日志
- 生成的 prompt
- 复制 prompt 按钮
- 输出目录或结果文件信息

日志可能较长，因此详情区需要支持滚动查看。

## 10. 功能需求

### 10.1 任务创建

- 用户可以一次选择一个或多个 `.docx` 文件
- 用户可以不上传关键词配置文件
- 没有域名或没有 Word 文件时不能提交任务
- 任务一提交就应出现在队列中

### 10.2 队列执行

- 所有任务按提交顺序依次执行
- 同一时间只执行一个任务
- 当前任务运行时，可以继续新增任务
- 当前任务失败时，不影响后续任务继续执行

### 10.3 本地文件处理

- Word 文件必须能直接从本地选择
- 关键词配置文件必须能直接从本地选择
- 页面层要把文件安全地交给本地处理逻辑

### 10.4 prompt 展示与复制

- 每个完成的任务都必须展示自己的 prompt
- 每个任务都必须有独立的复制按钮
- 复制某个任务的 prompt 时，不影响其他任务

### 10.5 日志展示

- 每个任务都要有独立日志
- 任务运行过程中，日志应可实时查看
- 任务完成后，日志仍然保留在页面中可查看

### 10.6 结果摘要

每个任务在完成后，摘要区应尽量展示：

- sitemap 抓取情况
- 产品页或集合页数量
- 博客页数量
- 成功处理的文章数
- 添加的关键词链接数
- 输出目录路径
- 最终任务状态

## 11. 非功能需求

- 工具运行在本地单机环境
- 启动尽量快
- 不依赖账号系统
- 第一阶段不强制引入数据库
- 错误提示尽量清楚
- 依赖尽量少，便于本地运行

第一阶段优先级排序如下：

1. 能用
2. 简单
3. 稳定
4. 再考虑结构是否更漂亮

## 12. 推荐技术方向

推荐采用：

- 本地网页界面
- 本地服务层
- 复用 Python 脚本逻辑

推荐目录分工：

- `ops_tool/scripts/`
  存放核心 Python 处理逻辑
- `ops_tool/web/`
  存放本地网页前端和界面相关代码
- `ops_tool/data/`
  存放可复用输入文件或配置
- `ops_tool/output/`
  存放生成结果

推荐这样做的原因：

- 现有 Python 逻辑已经可用
- 改动风险更小
- 上线速度更快
- 后面继续扩展前端时也更自然

### 13.1 任务输入

- 域名
- CMS 模式

### 13.2 任务状态

- 任务 ID
- 创建时间
- 当前状态
- 当前进度描述
- 日志内容
- 结果摘要
- prompt 内容
- 输出目录
- 错误信息

第一阶段可接受：

- 队列状态保存在内存中
- 最终结果保存在本地输出目录中

## 14. 错误处理要求

前端版必须把错误说清楚，至少覆盖以下类别：

- 域名不合法
- sitemap 抓取失败
- Word 转换失败
- 关键词配置文件不合法
- prompt 生成失败
- 其他未预期内部错误

用户在看到错误时，应该能立刻知道：

- 是哪个任务失败了
- 失败发生在哪一步
- 后续任务是否继续执行了

## 15. 第一阶段验收标准

满足以下条件即可认为第一阶段达到可开发标准：

- 可以打开一个本地网页完成任务提交
- 可以直接选择本地 Word 文件
- 可以直接选择本地关键词配置文件
- 任务按顺序执行
- 任务运行中可以继续新增任务
- 每个任务都有独立日志
- 每个任务都有独立 prompt 展示
- 每个任务的 prompt 都支持一键复制
- 失败任务不会阻塞后续任务
- 现有脚本版仍然可以继续单独运行

## 16. 后续开发前说明

本需求文档确认后，下一步建议不是立刻写代码，而是先输出一份实现计划文档，把开发拆成清晰步骤，例如：

- 先整理脚本调用入口
- 再做任务队列层
- 再接页面输入与状态展示
- 再补日志和 prompt 复制
- 最后做联调和人工验证

这样后续开发会更稳，也更容易逐步确认。
